# Skills

I think of skills like jobs on a development team.

When you hire a QA engineer, you don't explain WCAG 2.1 AA to them from scratch on every
project. They bring that knowledge with them. You just tell them which page to review.
Skills work the same way — they're the standing expertise the agent carries into any task,
loaded on demand, without you having to re-explain the fundamentals each time.

---

## The Mental Model

Each skill is a specialist. A specialist:

- Has a defined scope — they know what they're responsible for and what they're not
- Brings their own methodology — you don't micromanage how a QA engineer runs a checklist
- Only shows up when relevant — you don't invite the SEO advisor to a TDD session
- Defers to project context — a specialist working at your company follows your conventions,
  not just general best practices

The agent reads skill descriptions to decide who to bring in. If you say "audit this page for
accessibility", it loads `qa`. If you say "grill me on this plan", it loads `grill-me`. You
don't configure this — it happens automatically based on what you're asking for.

---

## Two Tiers

**Universal skills (live here):**
Role definition, methodology, checklists, and principles that apply to any project.
A universal skill should work without modification in any codebase, for any site, for any team.
If you'd have to edit it before using it on a new project — it's not universal yet.

**Project overlays (live in the project):**
The project-specific layer that extends a universal skill. Base URL, known issues, active
investigations, site-specific patterns, content backlog. An overlay doesn't redefine the role —
it gives the specialist their briefing notes for this particular job.

**Overlay file convention:** create `./skills/<name>.local.md` alongside the copied skill.
Do not edit `./skills/<name>/SKILL.md` directly — that is the universal base and will be
overwritten by `update.sh`. Using a separate `.local.md` file keeps base and overlay distinct
and protects your project context from being clobbered on updates.

Example — `seo` skill in agent-config:
```
You are the SEO advisor. Fix what's broken before optimizing what's working.
[Universal methodology, checklists, structured data patterns]
## Project Context
Check AGENTS.md or local skill overlays for site-specific configuration.
```

Example — `seo.local.md` overlay in a specific project:
```
Extends the universal SEO skill.
- Base URL: https://example.com
- Sitemap: auto-generated by vite.config.js on build
- Known issue: 12 pages currently unindexed — prioritize diagnosis before anything else
```

The agent reads both. The overlay wins on conflicts.

---

## What Belongs Here

A skill belongs in this repo if a developer on any project could drop it in and use it
immediately, without editing it first.

Ask: *would I need to change anything in this file before it's useful on a different project?*
- If no → it belongs here
- If yes → the parts that need changing belong in a project overlay, not in the skill itself

**Common ways a skill leaks project-specific content:**
- Hardcoded URLs, domain names, or property IDs
- References to specific files, tools, or build systems by name
- Known issues or active investigations for a particular site
- Content backlogs or gap analyses for a specific project

Strip those out. Put them in a `## Project Context` section with instructions to check the
project's AGENTS.md or local overlay. The skill stays universal. The project fills in the gaps.

## Adding a New Skill

1. Create `skills/<skill-name>/SKILL.md`
2. Write a frontmatter description that a new agent could use to decide whether to load it
3. Define the role clearly in the first paragraph — what does this specialist do?
4. Add methodology, principles, checklists, or examples
5. End with a `## Project Context` section pointing to AGENTS.md for project-specific config
6. Apply the "would I edit this?" test before committing

The description field is the most important part. It's what the agent reads to decide whether
this skill is relevant. Too vague → false triggers. Too narrow → missed triggers. Aim for
4–6 specific trigger phrases covering the most common ways someone would naturally ask for
this kind of help.
