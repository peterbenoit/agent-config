# OG Image Pipeline

OG images are typically generated by a build-time script or separate tool. The approach varies
by project — check AGENTS.md for the specific generator, commands, and file conventions.

## How It Works (General Model)

- A generator script reads page metadata and produces `*.png` files
- Images are copied into a public directory and referenced by `og:image` meta tags
- Incremental generation (only new/changed pages) is preferred over full regeneration
- AI-assisted generation is optional; default behavior should be deterministic templates
- If the generator is absent (e.g. CI), the build should warn but not hard-fail

## AI Generation Options

Common env vars used by OG generators with AI support:
- Enable/disable AI: `OG_AI_ENABLED`
- Provider and model selection: `OG_AI_PROVIDER`, `OG_AI_MODEL`
- Retry and timeout: `OG_AI_RETRIES`, `OG_AI_TIMEOUT_MS`
- Seed strategy: `OG_AI_SEED_STRATEGY` (controls determinism across runs)
- API key: check AGENTS.md for which key this project uses

## Review Checklist Before Publishing

1. Run a dry-run first and inspect planned outcomes before generating
2. Confirm prompts/seeds look deterministic and sane for changed pages
3. Generate images and verify cache behavior in logs (`cache-hit` vs regenerate)
4. Spot-check readability and accent consistency on sampled outputs
5. Confirm files are copied into the public directory and referenced by `og:image` tags
6. Check that `og:image` URLs are absolute (required by the Open Graph spec)

## Common Issues

- **Wrong image path:** `og:image` must be an absolute URL, not a relative path
- **Stale images:** Force-regenerate if template or metadata changed significantly
- **Missing images in CI:** Generator likely not present — confirm build still succeeds with warning
