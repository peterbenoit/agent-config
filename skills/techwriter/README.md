# techwriter — Deferred Skill

This folder is a placeholder. A `techwriter` skill has not been written because there's an
open question about whether it's distinct enough from skills that already exist.

---

## The Overlap Problem

Three skills currently cover adjacent territory:

- **`voice`** — prose quality, clarity, tone, directness. It's an editor. It reads your copy
  and makes it better.
- **`docs`** — documentation architecture. It decides what to write, how to structure it,
  and which document type (tutorial/how-to/reference/explanation) applies.
- **`508`** — accessibility compliance documentation (VPATs, remediation reports).

A `techwriter` skill would need to do something none of those do. The candidate territory:

- Writing for non-technical audiences (translating technical decisions for stakeholders)
- User-facing error messages and in-app copy (microcopy)
- Release notes and communications (different register than a changelog)
- Style guide enforcement across a large content surface

None of those are clearly needed yet.

---

## When to Write This Skill

Write `techwriter/SKILL.md` when you find yourself:

1. Regularly doing a specific task that neither `voice` nor `docs` covers well
2. Able to describe the role in one sentence without it sounding like a remix of the other two
3. Working on a project with a large enough content surface that a dedicated specialist makes sense

Until then, use `voice` for prose quality and `docs` for information architecture.

---

## If You Write It

The strongest differentiator would be **audience translation** — explaining technical things
to non-technical stakeholders, writing release communications, or managing a content style guide.
That's genuinely a different job than what `docs` does.

Trigger phrases that would distinguish it from existing skills:
- "write this for a non-technical audience"
- "release announcement"
- "explain this to stakeholders"
- "microcopy"
- "style guide"
